## 0. 結論

**MSERベースの検出カスケード（MSER → サイズフィルタ → NMS → 横マージ → アスペクト比フィルタ）を、連結成分（Connected Components、以下CC）ベースのパイプラインに置き換える。**

- CC方式は「二値化後のインクピクセルは必ずいずれかのCCに属する」ため、**二値化の成功を前提とすれば検出漏れが構造的に起きにくい**（二値化が新たな単一依存点になる — 節2-A・節4で明示的に扱う）
- 「タ」「ー」消失の正確な段階は未確定（候補4仮説 — 節1-3）だが、**CC設計は4仮説のどれが真でもすべて潰す**ため、刷新の判断は仮説の決着を待たない。決着自体は段階1の計装で安価につける
- MSERの重複bbox（約50個/文字）が存在しないため**NMSが丸ごと不要**になり、キメラbbox問題も消滅する
- 未解決だった**枠線リーク問題（候補A/B）を正確に解決**できる（CCピクセル描画方式）
- 追加依存ゼロ（OpenCVのみ）・処理時間は1〜2秒程度に短縮**見込み**（仮説。節5の実測指標で検証する）

**切替は評価ハーネスの数値で決める。** 段階1で現行MSER版のベースラインを計測し、CC版が全サンプルで上回った場合のみ切り替える。それまで現行コードは温存し、即時復帰できる状態を保つ（節6）。

## 1. 現状の根因分析

### 1-1. 観測されている事実（タスク.md・debug_bbox.png より）

| 観測 | 確度 |
|---|---|
| 漢字「初期化」「検証」は95〜96%で検出・認識OK | 事実 |
| 「タ」「ー」は最終出力にbboxが存在しない | 事実（debug_bbox.png） |
| 「ス」と「タ」をまたぐ複合bboxが存在する | 事実（debug_bbox.png — NMSキメラの状況証拠） |
| 矢印（↓）が62%でテキスト誤検出 | 事実 |
| 処理時間 9.3〜20秒 | 事実 |

### 1-2. MSERの原理的不適合（漢字○・カタカナ×の説明）

MSERは「明るさ閾値を変化させても**面積が安定している領域**」を検出する（本来は写真向け）。密な漢字は塊として安定するが、囲み領域のない疎なカタカナの細ストロークは閾値変化で面積が揺らぎ、安定領域になりにくい。観測の「漢字○・カタカナ×」パターンと整合する。「ス」「ト」が通り「タ」が落ちるのは本質的な差ではなく、**どちらも崖っぷちで閾値の僅かな揺らぎが明暗を分けている**疑いが強い。

### 1-3. 「タ」「ー」消失の候補仮説（4つ）— 段階1の計装で確定する

**重要な訂正**: 「bboxが最終出力にない」ことは「MSERが検出しなかった」ことを意味しない。最終出力の `text_blocks` はOCRが非空テキストを返した領域のみを含むため、消失はパイプラインの**どの段階でも**起こりうる。

| # | 仮説 | 機序 | 検証方法（段階1） |
|---|---|---|---|
| H1 | MSER未検出 | 細ストロークが安定領域にならない（1-2） | MSER直後のbboxオーバーレイ画像 |
| H2 | フィルタ足切り | サイズ（min_h=20・4xスケール。ーは16〜32pxで境界線上）またはアスペクト比（h<15 かつ w/h>10） | 各フィルタ前後のオーバーレイ |
| H3 | NMSキメラ吸収 | 面積優先NMSが「ス＋タ」複合領域を残し、個別bboxをIoU>0.3で抑制（複合枠は画像で確認済み） | NMS前後のオーバーレイ |
| H4 | **OCR空振り脱落** | 検出はされたが、単独クロップのOCRが空文字を返しtext_blocksから消えた | 切り抜き画像とOCR生結果のダンプ |

**マージ条件（height_ratio≥0.8）の正確な位置づけ**: マージはbboxを統合するだけで削除しないため、消失の直接の犯人にはなれない。ただし**間接経路**がある — 「ー」(h≈24)と「ス」(h≈180)は高さ比0.13で構造的に結合できず、仮にH1〜H3を生き延びても「ー」は**単独クロップ**としてOCRに送られる。文脈なしの「ー」一文字はTesseractが空文字・記号を返しやすく、H4に合流する。つまり現行設計では「ー」は検出されても認識段階で脱落する公算が大きい。

→ **どの仮説が真でもCC設計は対処済み**: H1=CCは検出を取りこぼさない / H2=固定閾値を文字サイズ自動推定に置換 / H3=NMS自体が消滅 / H4=グルーピングをOCRの**前**に行い、「ー」は「スタート」行クロップの一部として文脈付きで認識される。

### 1-4. NMSの面積優先がキメラbboxを作る

`_nms_regions` は面積大を優先して残すため、複数文字をまたぐ複合領域が個別文字bboxを抑制する。文字検出のNMSで「面積最大優先」は誤り（大きい＝正しいではなく、複数文字や枠線片の混入を意味しがち）。

### 1-5. 矢印誤検出は検出段階だけでは解決できない

矢印はインクであり文字とサイズが近い。bbox形状だけでの排除は困難で、最終防壁は認識信頼度だが、現在 `CONFIDENCE_THRESHOLD = 0` のため62%が素通りしている。

### 1-6. 処理時間の主犯

1920×1080を4倍（約33Mピクセル）にINTER_CUBICで拡大し、その全面にMSERを実行している。拡大が必要なのは認識に渡す小さなクロップだけであり、検出のための全画像拡大は不要。

### 1-7. ドキュメントとコードの乖離（要修正）

| 箇所 | ドキュメント | 実装 |
|---|---|---|
| ロジック解説.md Step 1-9 / ソース構造.md | 「2倍+Otsu → 4倍+Otsu → 2倍+Adaptive のfallbackフロー（現行実装）」 | `preprocess.py` に該当実装は**存在しない** |
| ロジック解説.md Step 1-7 | 「THRESH_BINARY_INV（反転）でテキストを白にする」 | `binarize_otsu` は **THRESH_BINARY（非反転）**。OCR用には実装側が正 |

採用後の改訂で解消する（節7）。

## 2. 確定パイプライン

### 2-0. 設計原則 — 失敗モードを「回復可能な側」に寄せる

> 二値化後のインクピクセルは、必ずいずれかの連結成分に属する。

CC方式では、**二値化が成功している限り**検出漏れが構造的に起きない。問題は「検出」から「二値化の品質」と「分類」（このCCは文字か図形か）に移る。

- **検出漏れ**（現行の失敗モード）: 下流で回復不能。bboxが無ければOCRも信頼度判定も走らない
- **分類ミス**（本設計の失敗モード）: あいまいクラス → 行文脈 → 信頼度ゲートの多段で回復できる

二値化が新たな最重要依存点になることは設計上明示する（A・節4）。

### 2-1. 全体図

```
INPUT (BGR)
↓ A. インク抽出: gray = min(B,G,R) → Otsu二値化（インク率検査で失敗検知 → Adaptive）
↓ B. 連結成分分析 cv2.connectedComponentsWithStats（8近傍・1倍スケール）
↓ C. CC分類（2パス）: ノイズ / 図形 / 文字候補 / あいまい
↓ D. 行グルーピング（中心バンド方式）+ あいまいCCの文脈確定
↓ E. 行ごとにCCピクセルのみを白キャンバスに描画して切り抜き
↓ F. 高さ正規化（文字高≈40px・INTER_CUBIC・余白10px）→ Tesseract PSM 7
↓ G. 信頼度ゲート（3帯域・保留は複数CC行のみ）
→ text_blocks（採用＋保留） + 予備マスク（採用のみ・ピクセル精度）
   └ Step 4 で保留を昇格/棄却 → 最終マスク再生成（Phase 2 のフック）
```

### A. インク抽出

```python
gray = np.min(image, axis=2).astype(np.uint8)  # min(B,G,R)
_, ink = cv2.threshold(gray, 0, 255, cv2.THRESH_BINARY_INV + cv2.THRESH_OTSU)
```

- `min(B,G,R)`: 青ペン・赤ペン・黒いずれも「白背景より少なくとも1チャンネルは暗い」性質だけを使う。色相に依存しないため「色除去を行わない」確定方針と矛盾しない。サンプル（青インク）では輝度変換よりコントラストが大きい
- **二値化失敗の検知（fallback発動条件）**: 二値化は本パイプラインの単一依存点なので、成否を安価に検査してからCCに進む

| 検査 | 初期値 | 判定 |
|---|---|---|
| インク率 = インク画素数 / 全画素数 | 0.05%未満 または 25%超 | 失敗とみなし Adaptive で再二値化 |

- 複数の二値化候補をOCR信頼度で競わせる案は**不採用**（Tesseract呼び出しが候補数倍になり速度利点を打ち消す。インク率検査は統計量のみで実質ゼロコスト）。低信頼行のみの局所再二値化リトライは段階3で要否判断

### B. 連結成分分析

- **1倍スケールで実行**（全画像拡大の廃止）。1920×1080の二値画像で数十ms
- 各CCの bbox・面積・ピクセルマスク（labels配列）を取得し、以降の全段で使い回す

### C. CC分類（2パス）

文字サイズ推定が分類に要り、分類が文字サイズ推定に要る循環を避けるため2パスで行う。

**パス1（画像規模の明らかな図形とノイズを除く）**:

| # | 条件 | 分類 | 根拠 |
|---|---|---|---|
| 1 | 面積 < 8px | ノイズ | ゴミ・かすれ点 |
| 2 | w > 0.25×画像幅 または h > 0.25×画像高 | 図形 | 画像規模のCC（大外枠・長い区切り線） |

残りのCCの高さの**中央値**を暫定の文字サイズ推定値とする（固定値でなくデータから推定 — 解像度・ペン太さに追従）。

**パス2（文字サイズ基準の判定）**:

| # | 条件 | 分類 | 根拠 |
|---|---|---|---|
| 2' | 密度 < 12% **かつ** max(w,h) > 3×文字サイズ | 図形（中空枠） | ノード枠（例: フロー.pngの280×120px箱 = 画像幅の15%）はルール2を通過するが、中空（インク率8〜10%）かつ文字より大きい。文字はどれだけ大きくても密度がこれより高い |
| 3 | 細長度 > 8 **かつ** 長辺 > 2×文字サイズ | 図形 | 長い接続線。**2条件は必ずAND**。「ー」「｜」「1」「(」等は長辺≈1文字サイズのため該当しない |
| 3' | 細長度 > 8 だが 長辺 ≤ 2×文字サイズ | **あいまい** | 「ー」「｜」「(」等の細長グリフと短い線分。Dの文脈で確定 |
| 4 | 上記以外 | 文字候補 | |

密度 = インク画素数 / bbox面積。「ー」は密度が高く（バーそのものがbboxを満たす）ルール2'に該当しない。

**分類で確定させないものを確定させない**: 矢じり付き矢印（bbox約30×90px・細長度≈3・密度≈20%）はどのルールにも掛からず文字候補に紛れ込む。これは設計上**許容**する — D（孤立判定）とG（単独CC行は保留なし）の二段で落とす。分類段階で無理に閾値を増やすより、後段の文脈で落とす方が誤爆が少ない。

重み付きスコア分類（例: 0.4×密度+0.3×近傍+0.3×OCR信頼度）は**不採用**: (1) 重み自体が根拠のない新しいマジックナンバーで教師データがない (2) OCR信頼度の分類利用は循環参照（分類がOCR対象を決める）。評価ハーネスで境界事例の失敗が数値化されたら、その時のデータでスコア式・学習分類器を再検討する。

### D. 行グルーピング（中心バンド方式）

現行マージの `height_ratio >= 0.8` は「ー」を単独クロップに追い込む（節1-3 H4）ため**使わない**。

| 条件 | 式 | 根拠 |
|---|---|---|
| 水平ギャップ | gap ≤ 0.8 × 行高さ | 文字間隔は文字高さより小さい |
| 垂直中心ずれ | &#124;cy(CC) − cy(行)&#124; ≤ 0.4 × 行高さ | 「ー」は高さ6pxでも**中心位置**は行バンドの中央にある |

- 高さ比でなく中心位置で判定するため「ー」(h≈6px)と「ス」(h≈45px)が同じ行に束なり、**OCRは「スタート」を一行として文脈付きで読む**（H4対策の本体）
- 重複bboxが存在しないためNMSは不要。推移的マージは現行同様
- **あいまいCC（ルール3'）の確定**: 文字候補CCと同じ行に束なった → 文字 ／ 孤立 → 図形（OCRに渡さない）。「文字は行を成し、接続線は孤立する」というダイアグラムの構造的性質を使う

### E. CCピクセル描画による切り抜き — 枠線リーク問題（候補A/B）の正確な解

行に属する**CCのピクセルだけ**を白キャンバスに描画してから切り抜く。

```python
canvas = np.full((ch, cw), 255, dtype=np.uint8)
for cc_id in line_member_ids:
    canvas[labels_crop == cc_id] = 0
```

- 枠線は別CCなので、座標がどれほど近くても**物理的に混入し得ない**
- 候補B（モルフォロジー線除去）の「一」「口」を壊すリスクもない。**候補A/B論争はこの方式で解消**

### F. 認識（Tesseract）

- 行クロップの文字高が約40pxになるよう拡大（安定域30px以上・INTER_CUBIC）。**拡大するのは行クロップだけ**（処理時間の主因を除去）
- 周囲に10pxの白余白（Tesseractは画像端の文字が苦手）
- `--psm 7`・`lang="jpn"` は現行を継承。入力は既にノイズ除去済みの二値画像

### G. 信頼度ゲート（3帯域・保留は複数CC行のみ）

Tesseractの信頼度は不安定（同品質入力でも±10程度揺れる）なため単一閾値にしない:

| 帯域 | 複数CCの行 | 単独CCの行 |
|---|---|---|
| conf ≥ 70 | **採用** | **採用** |
| 50 ≤ conf < 70 | **保留**（`status: "provisional"` でtext_blocksに残す） | **棄却** |
| conf < 50 | 棄却 | 棄却 |

- **保留を複数CC行に限定する理由**: 行を成すのはほぼ確実にテキスト（「スタート」全文字60前後のケースを救済）。一方、分類をすり抜けた図形片（矢じり付き矢印など）は単独CCで現れるため、保留の対象外にすることで誤保留がマスクや構造データを汚す経路を遮断する
- 保留の最終判断はStep 4の空間関係で行う（例: ノード枠内側の保留ブロックはほぼ確実にテキスト → 昇格）
- 単独CCの孤立テキスト（例: 間取り図の「北」一文字）がconf 50〜69で棄却される副作用は**既知のトレードオフ**として受け入れ、評価ハーネスで頻度を監視する
- 認識結果が空文字・記号のみは帯域に関わらず棄却。帯域境界（50/70）は初期値で、評価ハーネスで調整

### マスク生成 — 予備マスクは「採用」のみ（ピクセル精度）

```python
mask = np.isin(labels, adopted_text_cc_ids).astype(np.uint8) * 255
mask = cv2.dilate(mask, np.ones((3, 3), np.uint8), iterations=2)
```

- **bbox矩形でなくCCピクセル＋2〜3px膨張**: 矩形マスクは文字が枠線に近い場合（「スタート」）に枠線まで消し、Step 3の輪郭を分断する。ピクセルマスクなら図形線を傷つけない
- **保留はマスクに含めない**。失敗の非対称性で判断する:
  - 保留が実は図形片（矢印）だった場合にマスクすると → 図形線が欠損し、**構造データから接続が消える（回復不能）**
  - 保留が実はテキストだった場合にマスクしないと → 文字ストロークがStep 3に漏れ小さな輪郭ノイズになる（Step 3は手書きのカスレ・点ノイズ対策として最小サイズフィルタを持つ必要がどのみちあり、**そこで吸収可能**）
  - 回復不能な側を避ける
- **Step 4 フィードバックフック（Phase 2）**: Step 4で保留が昇格したら最終マスクを再生成し、必要に応じて図形抽出を部分再実行する。Phase 1ではフックの口（保留ブロックのCC情報保持）だけ用意し、実装しない

### CC内部表現の保持（Step 4への布石）

検出過程で各CCの bbox・重心・密度・近傍CC・所属行・OCR結果が自然に揃う。`detect.py` の内部表現として破棄せず保持する（公開APIには載せない。text_blocks / mask は不変）。Step 4の距離・包含計算と保留昇格判定がこのグラフをそのまま使える。

### 巨大CC融合の検知（観測フックのみ・Phase 1）

文字ストロークが枠線に物理接触すると1つのCCに融合し、図形側に分類されて文字が消える。本質的対策（局所再分割）はPhase 1の対象外だが、**起きたことに気づける仕組み**だけ先に入れる:

- 図形分類されたCCのbbox内部に文字候補CCが**1つもない**場合、`empty_frame` 警告をdebug出力に記録（フローチャートのノードは通常ラベルを含むため、空フレームは融合の兆候）
- 評価ハーネスの監視指標に空フレーム数を含める
- 将来の局所再分割パスは「図形CCのピクセルマスク」を入力に取る設計とし、CC内部表現の保持（上記）がそのまま入力口になる

## 3. 現行 vs 確定案 比較表

| 観点 | 現行（MSERカスケード） | 確定案（CCパイプライン） |
|---|---|---|
| 検出漏れ | MSERの安定性に依存（タ・ー消失） | **二値化成功を前提に構造的に起きにくい** |
| 「ー」の認識文脈 | 単独クロップ（H4: 空振り脱落リスク） | 「スタート」行の一部として文脈付き認識 |
| 重複bbox | 約50個/文字 → NMS必須（キメラ問題） | **0個（NMS不要）** |
| 枠線リーク | 未解決（候補B保留中） | CCピクセル描画で**正確に解決** |
| マスク品質 | 矩形（図形線を破壊しうる） | ピクセル精度・採用のみ（図形線無傷） |
| 矢印誤検出 | bbox形状のみでは排除困難 | 孤立CC判定＋単独CC行は保留なし、の二段 |
| 処理時間 | 9〜20秒（全画像4倍拡大＋MSER） | 1〜2秒見込み（**実測で検証** — 節5） |
| パラメータ | 約12個の固定値（1枚に手調整） | 約7個＋文字サイズは自動推定 |
| 追加依存 | — | **ゼロ**（OpenCVのみ） |
| 縦書き | 非対応 | 非対応（同等。Phase 3以降） |

## 4. リスクと限界（正直な評価）

| リスク | 内容 | 対応 |
|---|---|---|
| **二値化が単一依存点になる** | 文字が二値化で背景側に落ちるとCC以前に消える。「漏れない」保証は二値化成功が前提 | インク率検査→Adaptive fallback（節2-A）。**評価セットに紙の写真サンプルを必ず含め常時監視**。根本的には「間取り図の入力品質（A〜D）」の決定が先決 |
| **文字と枠線の物理接触（CC融合）** | 融合CCが図形分類され文字ごと消える | Phase 1: empty_frame警告で**検知**（節2）。再分割はCCピクセルマスクを入力とする将来パスとして口だけ確保。現行方式でも読めないケースであり退行ではない |
| **透視歪み（紙の写真）** | 斜め撮影では行グルーピングの水平前提が崩れる | Phase 1対象外（現サンプルは歪みなし）。透視補正は四隅検出という別の難問（既知課題#5）のため、入力品質決定後に位置づける |
| **手書き日本語認識の天井** | 検出を直しても崩れた手書きでTesseract（活字学習）が限界になる可能性 | Recognizerをインターフェース分離。**判断基準: 整った手書きで認識一致率 < 80% なら PaddleOCR（ONNX: 検出+認識 約15〜30MB）を評価**。EasyOCR（数百MB）より軽くスマホ500MB制限と整合 |
| **孤立単独CCテキストの棄却** | 「北」等の一文字注記がconf 50〜69だと消える | 既知のトレードオフ（節2-G）。評価ハーネスで頻度を監視し、問題なら帯域境界を調整 |
| **処理時間1〜2秒は仮説** | Tesseract呼び出し回数次第で超過しうる | 節5の実測指標（平均/95%/最大/OCR回数）で検証。仮説のまま確定扱いしない |

## 5. 評価ハーネス（パイプライン刷新と同時に導入・最優先）

現状の「フロー.png 1枚への手調整」は精度が頭打ちになる構造要因（1枚に過適合し、別サンプルの破壊に気づけない）。

1. `samples/` にドメイン別サブフォルダ（`flow/`・`floor/`・`circuit/`）で3〜5枚。**紙の写真を必ず1枚以上含める**（二値化依存点の監視 — 節4）
2. 各サンプルに期待値ファイル:

```json
// samples/flow/フロー.expected.json
{ "texts": ["スタート", "初期化", "検証"] }
```

3. `tests/eval_ocr.py`（新規）: 全サンプルを回し画像ごとに表で出力:

| 指標 | 定義 |
|---|---|
| 検出再現率 | 期待テキストのうちbboxが取得できた割合 |
| 誤検出数 | テキストでない領域の検出数（矢印など） |
| 認識一致率 | 検出できた領域のうちテキストが正しく読めた割合 |
| 処理時間 | 平均 / 95パーセンタイル / 最大（秒） |
| OCR呼び出し回数 | Tesseract起動回数（処理時間の支配項の直接監視） |
| 空フレーム数 | ラベルを含まない図形CC数（CC融合の兆候監視） |

4. **変更のたびに全サンプルで実行**し数値で判断（目視debug画像は補助に格下げ）

## 6. 段階的移行計画

公開API（`text_blocks` / `mask` の形式）は不変のため `test_ocr.py` と将来のStep 3はそのまま動く。各段階が独立に検証可能。

| 段階 | 内容 | 完了条件 |
|---|---|---|
| 1 | eval harness + 期待値ファイル + **現行パイプラインの段階別計装**（MSER直後/各フィルタ後/NMS後/マージ後のbboxオーバーレイ画像、クロップとOCR生結果のダンプ） | 現行ベースライン数値の固定 ＋ **タ・ー消失段階の確定（H1〜H4の決着）** |
| 2 | `detect.py` 新規実装（A〜D + 段階別デバッグ出力） | 検出再現率: フロー.pngで「スタート」4文字を含む全文字のbbox取得 |
| 3 | E〜G接続。`run_ocr_with_preprocess` の内部だけ差し替え（シグネチャ不変） | 認識一致率がベースライン以上 |
| 4 | ピクセルマスク化（採用のみ）+ CC内部表現・empty_frame警告 | debug_masked.png で図形線無傷を確認 |
| 5 | MSER系コード削除・ドキュメント全面改訂 | **全サンプルで旧版以上の数値**（満たさなければ削除しない） |

段階1の計装は約30分の作業で、H1〜H4のどこで消えたかを直接確定できる。「MSER+新グルーピング」「CC+旧グルーピング」のようなハイブリッド構成の分離計測は**不採用** — マージはbboxを削除しないため計装で攻めれば帰属は付き、ハイブリッドのためのアダプタ実装（bboxリストとCCピクセル表現の相互変換）はコストに見合わない。

### 第2巡（2026-06-12・破綻候補の指摘）

| 指摘 | 判断 | 要点 |
|---|---|---|
| ① 真犯人はグルーピング（height_ratio）かも。検出器×グルーピングの2×2分離計測を | **意図を採用・手段は変更** | マージはbboxを削除しないため「グルーピングが直接の犯人」は機序として成立しない。ただし**間接経路（マージ失敗→単独クロップ→OCR空振り脱落=H4）を見落としていたのは原案の過信**であり、消失段階は未確定と訂正（節1-3）。決着は2×2ハイブリッド（アダプタ実装が高コスト）でなく**段階1の段階別計装**（約30分）でつける。CC設計はH1〜H4のどれが真でも対処済みのため、刷新判断は決着を待たない |
| ② CC融合（文字＋枠が1CC化）の逃げ道 | **採用（検知のみ）** | empty_frame警告＋再分割パスの入力口（CCピクセルマスク保持）を用意。実装は後回しで可、の提案通り。検討過程で**原案の分類の穴を自己発見**: フロー.pngのノード枠（画像幅の15%）はルール2（>25%）を通過してしまう → 中空判定ルール2'（密度<12% かつ >3×文字サイズ）を追加 |
| ③ 保留帯域をマスクに入れると図形欠損 | **採用（第1巡の自判断を撤回）** | 矢じり付き矢印のbboxは細長度≈3・密度≈20%で分類網を通過しOCRに到達しうると計算で確認 — 「保留が図形片である可能性は低い」という第1巡の根拠は崩れた。マスク=採用のみ＋**保留は複数CC行限定**（単独CC行は保留なし）の二重防御に変更。失敗の非対称性（図形欠損は回復不能・テキスト漏れはStep 3のノイズフィルタで吸収可能）を判断基準とした |
| ④ 二値化単一依存点・写真サンプル必須 | 既反映を強化 | 評価セットへの写真サンプル投入を必須要件として明文化（節5） |
| ⑤ 1〜2秒は仮説 | 採用 | 「見込み」と明記し、平均/95%/最大/OCR回数の実測指標を節5に追加 |
