## 目的

手書きで描いた図形および付随する文字を、**構造データ**に変換する。SVGはその構造データの出力先の1つに過ぎない。

### ターゲット用途

- フローチャート
- 回路図
- 建築の間取り図
- ダイアグラム全般

### 既存ツールとの差別化

Adobe IllustratorのトレースやGeminiとの違いは「形をなぞるだけでなく、**テキストと図形の空間的関係を理解した構造化データを生成する**」点にある。

```xml
<!-- 非セマンティック（AIトレース・Illustratorの出力） -->
<rect x="225" y="40" width="150" height="60"/>
<text x="300" y="70">スタート</text>  <!-- rectと無関係 -->
<line x1="300" y1="100" x2="300" y2="130"/>  <!-- 何と何をつなぐか不明 -->

<!-- セマンティック（GraphicCopyが目指す出力） -->
<g id="node-start" data-type="process">
  <rect x="225" y="40" width="150" height="60"/>
  <text data-label-for="node-start">スタート</text>
</g>
<g id="edge-1" data-from="node-start" data-to="node-init">
  <line x1="300" y1="100" x2="300" y2="130"/>
</g>
```

| 操作 | 非セマンティック | セマンティック |
|---|---|---|
| ノードを移動 | rectとtextを別々に動かす | グループごと動かすだけ |
| 接続関係を取得 | 座標から推測 | `data-from/to`から確実に取得 |
| 他フォーマットへ変換 | 座標推測が必要 | 構造から直接変換可能 |

## Step 1: OCR・テキスト検出

> **設計ステータス**: 本節は CC パイプライン確定設計（2026-06-12、[fable 5提案.md](fable%205提案.md)）を記述する。段階2〜5の実装完了まで現行 MSER コードは温存される（eval-gated 移行）。

### 処理フロー

Tesseractは全体画像から小さいテキスト（10〜14px）の領域を直接見つけられないが、該当箇所を切り抜いてOCRすると読める。問題はOCRの読解能力ではなく、**どこを切り抜くか・どう行にまとめるか**にある。CCでその「どこ」と「まとめ方」を解決する。

```
全体画像（BGR）
↓ 1-A. インク抽出: min(B,G,R) → Otsu二値化 + インク率検査（失敗 → Adaptive）
↓ 1-B. 連結成分分析: cv2.connectedComponentsWithStats（8近傍・1倍スケール）
↓ 1-C. CC分類（2パス）: ノイズ / 図形 / 文字候補 / あいまい
↓ 1-D. 行グルーピング（中心バンド方式）+ あいまいCCの文脈確定
↓ 1-E. CCピクセル描画で行クロップ生成
↓ 1-F. 高さ正規化（文字高 ≈ 40px）→ Tesseract PSM 7
↓ 1-G. 信頼度ゲート（3帯域）
→ text_blocks（採用＋保留）
```

---

### Step 1-A. インク抽出

**min(B,G,R) グレースケール化**

```
インクチャンネル強度 = min(B, G, R)
```

標準輝度変換（B×0.114 + G×0.587 + R×0.299）は青ペン（B↑G↑R↓）のインクが白背景と近い値になりコントラストが落ちる。`min(B,G,R)` は「白背景より少なくとも1チャンネルは暗い」という性質だけを使うため、黒・青・赤いずれのインクも高コントラストで抽出できる。色除去を行わない設計原則とも矛盾しない。

**Otsu二値化（THRESH_BINARY_INV）**

```python
gray = np.min(image, axis=2).astype(np.uint8)
_, ink = cv2.threshold(gray, 0, 255, cv2.THRESH_BINARY_INV + cv2.THRESH_OTSU)
```

`THRESH_BINARY_INV`: インクピクセル（暗）→ 255（白）、背景（明）→ 0（黒）。この向きにすることで `cv2.connectedComponentsWithStats` が非ゼロ（白=インク）のブロブを連結成分として検出できる。Otsu法は画像全体のヒストグラムから背景とインクを最もうまく分離する閾値を自動計算する。

**インク率検査（fallback 発動条件）**

二値化はパイプライン全体の単一依存点。成否を安価に検査してから次段へ進む。

| 検査 | 判定 |
|---|---|
| インク率 = インク画素数 / 全画素数 が 0.05% 未満 | 二値化失敗（ほぼ空白・閾値ミス）→ Adaptive で再二値化 |
| インク率 = インク画素数 / 全画素数 が 25% 超 | 二値化失敗（インクと背景が反転・ノイズ過多）→ Adaptive で再二値化 |

インク率検査は統計量のみで実質ゼロコスト。Tesseract呼び出しで複数の二値化候補を競わせる方式は採用しない（Tesseract起動が候補数倍になり速度利点を打ち消す）。

**Adaptive Threshold（fallback）**

Otsu失敗時に発動する。各ピクセルの周辺領域（ブロックサイズ = 短辺の1/10程度）の平均を基準に閾値を決めるため、照明ムラ・影に強い。ただし背景のムラも「インク」として拾うノイズリスクがあるため、常用でなくfallbackとして位置づける。

**（参考）HSV色空間**

BGRは明暗の影響を受けるが、HSVのH（色相: 0〜180）は色の種類だけを表すため照明条件に左右されない。色ベースの前処理が必要になった場合に利用。現行では使用しない。

### Step 1-B. 連結成分分析

```python
num_labels, labels, stats, centroids = cv2.connectedComponentsWithStats(ink, connectivity=8)
```

- **8近傍連結**: 斜め方向も連続とみなす。4近傍では「×」のような斜め線が複数CCに分裂する
- **1倍スケールで実行**: 4倍拡大は不要。全画像4倍拡大→MSER全面実行は処理時間の主因だった。CC分析は1920×1080の二値画像で数十ms
- `stats` 配列から各CCの bbox（左上 x,y・幅・高さ）・面積（インク画素数）を取得
- `labels` 配列は元画像と同サイズで、各ピクセルがどのCC番号に属するかを示す。後の CCピクセル描画・マスク生成で直接使用する
- `centroids` 配列から各CCの重心座標を取得。行グルーピングの中心バンド判定に使用

> CC方式では「二値化後のインクピクセルは必ずいずれかのCCに属する」ため、二値化の成功を前提とすれば検出漏れが構造的に起きにくい。従来のMSERカスケードでは「安定領域として検出されなかったCCがそもそも候補に上がらない」という下流で回復不能な漏れが発生していた。

### Step 1-C. CC分類（2パス）

分類に文字サイズが要り、文字サイズ推定には「文字候補」の集合が要る、という循環を避けるため2パスで行う。

**パス1（画像規模の明らかな図形とノイズを除く）**

| # | 条件 | 分類 | 根拠 |
|---|---|---|---|
| 1 | 面積 < 8px | ノイズ | ゴミ・かすれ点 |
| 2 | w > 0.25 × 画像幅 **または** h > 0.25 × 画像高さ | 図形 | 大外枠・長い区切り線 |

パス1を通過した残りCCの高さの**中央値**を暫定の文字サイズとする。固定値でなくデータから推定するため、解像度・ペン太さに自動追従する。

**パス2（文字サイズ基準の判定）**

以下の指標を使う:
- 密度 = インク画素数 / bbox面積
- 細長度 = max(w, h) / min(w, h)

| # | 条件 | 分類 | 根拠 |
|---|---|---|---|
| 2' | 密度 < 12% **かつ** max(w,h) > 3 × 文字サイズ | 図形（中空枠） | ノード枠はルール2（>25%）を通過しうる（例: 画像幅15%の箱）。しかし中空（インク率8〜10%）かつ文字より大きい。「ー」等のグリフは密度が高くここに非該当 |
| 3 | 細長度 > 8 **かつ** 長辺 > 2 × 文字サイズ | 図形（接続線） | 長い線分。**2条件は必ずAND**。「ー」「｜」「1」「(」等は長辺 ≈ 1文字サイズのため非該当 |
| 3' | 細長度 > 8 **かつ** 長辺 ≤ 2 × 文字サイズ | **あいまい** | 「ー」「｜」「(」等の細長グリフと短い線分。Step 1-D の行文脈で確定 |
| 4 | 上記以外 | 文字候補 | |

矢じり付き矢印（bbox約30×90px・細長度≈3・密度≈20%）はどのルールにも掛からず文字候補に紛れ込む。これは設計上許容し、Step 1-D（孤立判定）と Step 1-G（単独CC行は保留なし）の二段で排除する。分類段階で無理に閾値を増やすより後段の文脈で排除する方が誤爆が少ない。

### Step 1-D. 行グルーピング（中心バンド方式）

旧実装の `height_ratio >= 0.8` 条件は、「ー」(h≈24px) と「ス」(h≈180px) の比が0.13となり構造的に同一行に束ねられない。「ー」は単独クロップとしてOCRに送られ、文脈のない「ー」一文字はTesseractが空文字・記号を返しやすい（結果、text_blocksから脱落する）。この問題を**中心バンド方式**で解消する。

| 条件 | 式 | 根拠 |
|---|---|---|
| 水平ギャップ | gap ≤ 0.8 × 行高さ | 文字間隔は文字高さより小さい |
| 垂直中心ずれ | &#124;cy(CC) − cy(行)&#124; ≤ 0.4 × 行高さ | 「ー」は高さ6pxでも**中心位置**は行バンドの中央にある。「ス」と同じ行に束なり「スタート」として文脈付きOCR |

高さ比でなく中心位置で判定するため、高さの異なる文字（大文字・ふりがな・記号）も自然に同一行に束なる。推移的マージ（変化がなくなるまで繰り返し）は現行と同様。

**あいまいCC（ルール3'）の文脈確定**:
- 文字候補CCと同じ行に束なった → **文字**として確定（OCRへ渡す）
- 孤立（隣接する文字候補がない）→ **図形**として確定（OCRに渡さない）

「文字は行を成し、接続線は孤立する」というダイアグラムの構造的性質を利用した判定。

### Step 1-E. CCピクセル描画による切り抜き

行に属するCCのピクセルだけを白キャンバスに描画してから切り抜く。

```python
canvas = np.full((ch, cw), 255, dtype=np.uint8)
for cc_id in line_member_ids:
    canvas[labels_crop == cc_id] = 0
```

**枠線リーク問題の解消**: 旧実装はPAD=10pxの余白に枠線ピクセルが混入し（枠線のCCとテキストのCCが空間的に近い場合）、Tesseractの文字セグメンテーションが乱れていた。CCピクセル描画方式では**枠線は別CCに属するため、座標がどれほど近くても物理的に混入しない**。以前に「候補B（モルフォロジー変換）」として検討していた枠線除去は不要になり、漢字「一」「口」等が誤って消されるリスクもない。

---

### Step 1-F. 認識（Tesseract）

Step 1-E で得た行クロップを正規化してから OCR に渡す。

- 文字高が約40pxになるよう拡大（安定域30px以上・**INTER_CUBIC**）。**拡大するのは行クロップのみ**（全画像拡大は不要。処理時間の主因を除去）
- 周囲に10pxの白余白（Tesseractは画像端の文字が苦手）
- `--psm 7`（1行として認識）・`lang="jpn"` は現行を継承。入力は既にノイズ除去済みの二値画像

**補間法 INTER_CUBIC**: 周囲16ピクセルを使用しエッジが滑らか。INTER_NEAREST はジャギーが増えるため非推奨。

---

### Step 1-G. 信頼度ゲート（3帯域）

Tesseractの信頼度は同品質入力でも±10程度揺れるため単一閾値にしない。

| 帯域 | 複数CCの行 | 単独CCの行 |
|---|---|---|
| conf ≥ 70 | **採用** | **採用** |
| 50 ≤ conf < 70 | **保留**（`status: "provisional"`） | **棄却** |
| conf < 50 | 棄却 | 棄却 |

**保留を複数CC行に限定する理由**: テキストの行は複数の文字CCが集まるため、行を成すのはほぼ確実にテキスト（「スタート」全文字が60前後になるケースを救済）。分類をすり抜けた図形片（矢じり付き矢印等）は単独CCで現れるため保留対象外にすることで、図形線がマスク・構造データを汚す経路を遮断する。

認識結果が空文字・記号のみは帯域に関わらず棄却。帯域境界（50/70）は評価ハーネスで調整する（初期値）。

**保留の最終判断**: text_blocksに `status: "provisional"` で含める。Step 4の空間関係推定で行う（例: ノード枠内側の保留ブロックはほぼ確実にテキスト → 採用昇格）。Phase 1ではフックの口だけ確保し実装しない。

**孤立単独CCテキストの棄却（既知のトレードオフ）**: 「北」等の一文字注記がconf 50〜69で棄却される副作用がある。評価ハーネスで頻度を監視し、問題なら帯域境界を調整する。

## Step 2: テキストマスク生成

### 処理フロー

```
Step 1 の text_blocks（採用のみ）
↓ 2-1. 採用CCのピクセルをマスクに書き込み
↓ 2-2. 2〜3px膨張（dilation）
→ mask（テキスト領域=白・それ以外=黒）
```

---

### Step 2-1. CCピクセルマスク生成

Step 1 で「採用」となったCCのピクセルを元画像スケールのマスクに書き込む。

```python
mask = np.isin(labels, adopted_text_cc_ids).astype(np.uint8) * 255
mask = cv2.dilate(mask, np.ones((3, 3), np.uint8), iterations=2)
```

**bbox矩形でなくCCピクセルを使う理由**: 矩形マスクは文字が枠線に近い場合（「スタート」の枠線外接など）に枠線ピクセルまで白塗りし、Step 3の輪郭を分断する。CCピクセルマスクは文字ストローク形状に沿うため図形線を傷つけない。

**保留はマスクに含めない（失敗の非対称性）**:
- 保留が実は図形片（矢印等）だった場合にマスクすると → 図形線が欠損し、**構造データから接続が消える（回復不能）**
- 保留が実はテキストだった場合にマスクしないと → 文字ストロークが Step 3 に漏れ小さな輪郭ノイズになる（Step 3 の最小サイズフィルタで**吸収可能**）

回復不能な側を避ける設計。

マスクがないと Step 3 でテキストの輪郭が図形として誤検出される。

### Step 3-6. 輪郭検出（findContours）

`cv2.findContours` が二値画像の白い塊の境界を追跡し、各塊のバウンディングボックス・面積・頂点を返す。形状に関係なく閉じた輪郭を追跡するため、L字・多角形・穴あり形状にも対応できる。

---

### Step 3-7. 制約付き形状認識

ユーザーが形状のヒントを与えた場合、対応するフィット関数を直接呼ぶことで精度を安定させる。ヒントなしでは「これは楕円か四角か」をアルゴリズムが確定できない。

| ヒント | アルゴリズム | 概要 |
|---|---|---|
| 楕円 | `cv2.fitEllipse()` | 輪郭点群に最小二乗フィットした楕円を返す |
| 円 | `cv2.minEnclosingCircle()` | 全輪郭点を含む最小の外接円を返す |
| 長方形 | `cv2.minAreaRect()` | 面積最小の外接矩形（回転あり）を返す |
| 多角形（n角形） | `cv2.approxPolyDP()` | Douglas-Peuckerアルゴリズムで頂点を間引いてn角形に近似 |
| 自由曲線 | Bézierフィット | 輪郭点群から制御点を推定してBézier曲線で近似 |

---

### Step 3-8. 非矩形・非凸形状の認識

| 形状 | 対応難度 | 使うアルゴリズム |
|---|---|---|
| 長方形 | ★☆☆ | `minAreaRect()` |
| L字・凸でない多角形 | ★★☆ | `approxPolyDP()` |
| H形・コの字（単純連結） | ★★★ | `findContours` + 凹み検出 |
| 中に柱がある部屋（穴あり） | ★★★★ | `findContours` の階層構造 |

`findContours` の階層構造（`RETR_CCOMP` / `RETR_TREE`）を使うと、輪郭の親子関係から「外壁の中に穴がある」構造を取得できる。

## Step 3': 数式・座標解析（数式系・将来）

### 処理フロー

数式系では Step 3（図形抽出）・Step 4（空間関係推定）は使わない。OCR で読んだ文字を起点に意味を解析し、そこから図形を生成する。

**図形系との本質的な違い**

| | 図形系 | 数式系 |
|---|---|---|
| 出発点 | 画像上の図形 | OCR で読んだ文字（数式） |
| テキストの役割 | 図形に紐付くラベル | 意味そのもの |
| 図形の扱い | 画像から抽出する | 意味解析の結果として生成する |

手書きの直線だけ見ても `y=2x` か `y=1.8x` か判断できない。しかし文字として `y=2x` が読めれば、傾き・切片・グラフ形状まで確定する。数式系は「文字 → 意味解析 → 図形生成」の順で処理する。

```
Step 1 の text_blocks（OCR結果）
↓ 3'-1. 数式・座標の識別
↓ 3'-2. 数式パース
↓ 3'-3. 構造データへの変換
→ 出力層へ
```

---

### Step 3'-1. 数式・座標の識別（実装時に追記）

OCR結果の各テキストブロックが「数式・座標」か「通常テキスト」かを判定する。

---

### Step 3'-2. 数式パース（実装時に追記）

数式の種類ごとに解析方法が異なる。

| 入力例 | 種別 | 解析結果 |
|---|---|---|
| `y = 2x` | 線形関数 | 傾き=2, 切片=0 |
| `y = x²` | 二次関数 | 係数・次数 |
| `y = sin(x)` | 三角関数 | 関数種別・係数 |
| `A=(1,2), B=(3,4)` | 座標データ | 点ラベルと座標値の対 |
| `x²+4x+4=0` | 方程式 | 解の情報（x=-2） |

---

### Step 3'-3. 構造データへの変換（実装時に追記）

```json
// y = 2x の場合
{ "type": "linear_function", "slope": 2, "intercept": 0 }

// A=(1,2) の場合
{ "type": "point", "label": "A", "x": 1, "y": 2 }

// x²+4x+4=0 の場合
{ "type": "equation", "expression": "x**2+4*x+4", "solutions": [-2] }
```
