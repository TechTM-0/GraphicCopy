## 目的

手書きで描いた図形および付随する文字を、**構造データ**に変換する。SVGはその構造データの出力先の1つに過ぎない。

### ターゲット用途

- フローチャート
- 回路図
- 建築の間取り図
- ダイアグラム全般

### 既存ツールとの差別化

Adobe IllustratorのトレースやGeminiとの違いは「形をなぞるだけでなく、**テキストと図形の空間的関係を理解した構造化データを生成する**」点にある。

```xml
<!-- 非セマンティック（AIトレース・Illustratorの出力） -->
<rect x="225" y="40" width="150" height="60"/>
<text x="300" y="70">スタート</text>  <!-- rectと無関係 -->
<line x1="300" y1="100" x2="300" y2="130"/>  <!-- 何と何をつなぐか不明 -->

<!-- セマンティック（GraphicCopyが目指す出力） -->
<g id="node-start" data-type="process">
  <rect x="225" y="40" width="150" height="60"/>
  <text data-label-for="node-start">スタート</text>
</g>
<g id="edge-1" data-from="node-start" data-to="node-init">
  <line x1="300" y1="100" x2="300" y2="130"/>
</g>
```

| 操作 | 非セマンティック | セマンティック |
|---|---|---|
| ノードを移動 | rectとtextを別々に動かす | グループごと動かすだけ |
| 接続関係を取得 | 座標から推測 | `data-from/to`から確実に取得 |
| 他フォーマットへ変換 | 座標推測が必要 | 構造から直接変換可能 |

## Step 1: OCR・テキスト検出

### 処理フロー

Tesseractは全体画像から小さいテキスト（10〜14px）の領域を見つけられないが、該当箇所を切り抜いてOCRすると読める。問題はOCRの読解能力ではなく、**どこを切り抜くかを見つけること**にある。MSERでその「どこ」を探す。

```
全体画像（BGR）
↓ 1-1. グレースケール変換（4倍拡大 → グレースケール）
↓ 1-2. MSER で文字候補bbox検出（サイズフィルタ含む）
↓ 1-3. NMS（重複除去・IoU > 0.3）
↓ 1-4. 横方向マージ（3条件）
↓ 1-5. アスペクト比フィルタ
   各領域を切り抜き
↓ 1-6. Resize + グレースケール変換（1-1と同様）
↓ 1-7. 二値化 → 1-8. Otsu法（失敗時は 1-9. Adaptive）
   Tesseract OCR（PSM 7）
```

---

### Step 1-1. グレースケール変換

カラー画像のB・G・R 3値を0〜255の1値に変換する。**MSERに渡す全体画像**と、**切り抜き後のOCR前処理（1-6）**の両方で使う。

```
グレー値 = B×0.114 + G×0.587 + R×0.299
```

係数は人間の視覚特性を反映（緑を最も明るく感じる）。合計は1.0。

MSERに渡すときは**二値化しない**。MSERは複数の明るさ閾値をまたいで塊を検出するアルゴリズムのため、先に二値化すると必要な明るさの情報が失われる。

---

### Step 1-2. MSER による文字候補領域検出

MSERはグレースケール画像で「明るさが安定している塊」を検出するアルゴリズム。文字領域は背景と明るさが異なるため安定した塊として検出される。

**スケール**: 全体画像を4倍拡大してから渡す。元画像10〜14pxの文字が40〜56pxになりMSERが検出しやすくなる。

**パラメータ**

| パラメータ | 値 | 意味 |
|---|---|---|
| `_min_area` | 150 | 検出する塊の面積下限（ピクセル数）。点ノイズを除外 |
| `_max_area` | 14400 | 面積上限。図全体のような大きすぎる領域を除外 |

**サイズフィルタ**（4倍スケール基準）: `min_h=20, max_h=320, min_w=10, max_w=800`

---

### Step 1-3. NMS（重複除去）

MSERは同一文字に対して明るさ閾値ごとに約50個の重複bboxを生成する。NMS（Non-Maximum Suppression）でIoUが閾値を超える重複を除去する。面積の大きい領域を優先して残す。

```
IoU（重複率） = 重複面積 / 合計面積
IoU > 0.3 → 後発のbboxを除去
```

---

### Step 1-4. 横方向マージ

同じ単語に属する複数のbbox（例:「初」「期」「化」）を1つにまとめる。以下の3条件を**全て**満たす場合のみ結合する。

| 条件 | 式 | 目的 |
|---|---|---|
| 高さ比 | `min(h1,h2) / max(h1,h2) >= 0.8` | 高さが大きく違う枠線と文字を結合しない |
| 縦重なり率 | `y_overlap / min(h1,h2) >= 0.7` | 縦方向にずれた枠線と文字を結合しない |
| 横ギャップ | `gap <= max(h1,h2)` | 文字高さ基準の相対値（拡大倍率変更に強い） |

推移的マージ（変化がなくなるまで繰り返し）をサポート。

**ログ出力（verbose=True）**

```
merge? h_ratio=0.91 overlap=0.82 gap=18 → True
merge? h_ratio=0.64 overlap=0.71 gap=12 → False (height_ratio)
```

**縦書き非対応**: 現実装は横書き専用。縦書き対応には `y_overlap_ratio` と横ギャップを縦方向に入れ替える必要がある。

---

### Step 1-5. アスペクト比フィルタ

「高さが極端に小さく かつ 幅/高さ比が大きすぎる」領域を枠線とみなして除外する。

条件: `高さ < 15` かつ `幅/高さ > 10.0` の**両方**を満たす場合のみ除外。

単独判定にしない理由: 長いラベル（「処理開始前チェック」等）は幅が長くても高さはそれなりにある。単独の幅/高さ判定では誤除外される。

---

### Step 1-6. Resize（切り抜き後のOCR前処理）

1-5 までで絞り込んだ各領域を切り抜き、Tesseractが認識できるサイズに拡大する。その後 1-1 と同様にグレースケール変換を行う。

Tesseractは文字の高さが約30px以上で精度が安定する。

**なぜ拡大が必要か**

文字が小さいほど輪郭のグレーピクセル（アンチエイリアシング）が占める割合が大きくなり、後の二値化で消えやすくなる。拡大すると内側の純粋な黒ピクセルが増え、アンチエイリアシングの影響が薄まる。

**補間法**: `INTER_CUBIC`（周囲16ピクセル使用・エッジがなめらか）。`INTER_NEAREST` はジャギーが増えるため非推奨。

---

### Step 1-7. 二値化

グレースケールのグラデーション（0〜255）を白（255）か黒（0）の2値に変換する。`THRESH_BINARY_INV`（反転）でテキストを白にするのは、輪郭検出が「白い塊」を探す仕様のため。

固定しきい値128を使うと、薄いグレーのテキストが背景と同じ側に分類されて消える。→ 1-8 の Otsu法で解決する。

---

### Step 1-8. Otsu法

画像全体のピクセル分布（ヒストグラム）から、背景とテキストを最もうまく分離できるしきい値を自動計算する。

**弱点**: 照明ムラ・影がある画像では全体の分布が偏り、適切なしきい値が選ばれないことがある。→ 1-9 の Adaptive Threshold にフォールバックする。

---

### Step 1-9. Adaptive Threshold

各ピクセルの周辺領域（例: 11×11px）の平均を基準にしきい値を決める。照明ムラ・影に強い。

局所基準のため、背景のムラも「テキスト」として拾うノイズリスクがある。Otsuで信頼度が低い場合のfallbackとして使う。

**fallback フロー（現行実装）**

```
2倍拡大 + Otsu → 信頼度低 → 4倍拡大 + Otsu → 信頼度低 → 2倍拡大 + Adaptive
```

---

### Step 1-10. 図形線とテキストの分離問題（候補A・B）

枠線に囲まれたテキストを読む際、枠線がbboxに食い込みTesseractの文字セグメンテーションが壊れることがある（「フロー.png」の「スタート」で確認済み）。1-7〜1-9 の前処理方針の選択に影響する。

**Step 1-5 で枠線 bbox を除外しても解決しない理由**

bbox は「ここに何かある」という**座標情報**に過ぎない。Step 1-5 でフィルタしても、除外されるのはその bbox がリストから消えるだけで、元画像のピクセルデータは一切変わらない。

```
元画像のピクセルデータ  ← 変わらない（枠線の実体）
      ↓
MSER が bbox を検出
      ↓
Step 1-5 で bbox をリストから除外  ← 座標情報が消えるだけ
      ↓
元画像のピクセルデータ  ← 変わらない（枠線は残っている）
```

そのため「スタート」のような枠線の内側にあるテキストを切り抜くとき、PAD=10px の余白分だけ周囲の画像も含まれる。枠線が近い場合、PAD の中に枠線のピクセルが入り込む。

```
切り抜き範囲（PAD込み）:

  ─────────────  ← 枠線のピクセルが入り込む
  │  スタート  │
  ─────────────  ← 枠線のピクセルが入り込む
```

この状態で Tesseract が読むと文字の上下に黒い横線があるように見えるため、文字セグメンテーションが乱れ「タ」「ー」が見失われる原因になりうる。

**候補A（現行実装）: グレースケール+Otsuのみ**

色除去は行わない。テキストが黒とは限らず、色でテキストと図形線を原理的に区別できないため。

```
BGR画像 → 1-6. Resize → グレースケール → 1-8. Otsu二値化 → Tesseract
```

懸念: Otsuのしきい値次第では枠線が「テキストと同じ側」に分類される可能性あり。

**候補B（未実装）: モルフォロジー変換による図形線除去**

「長さが大きく幅が細い連続した線」という形状的特徴で図形線を検出して除去する。色に依存しない。

```python
def remove_figure_lines(binary: np.ndarray) -> np.ndarray:
    h, w = binary.shape
    horizontal_kernel = cv2.getStructuringElement(cv2.MORPH_RECT, (w // 10, 1))
    horizontal_lines = cv2.morphologyEx(binary, cv2.MORPH_OPEN, horizontal_kernel)
    vertical_kernel = cv2.getStructuringElement(cv2.MORPH_RECT, (1, h // 10))
    vertical_lines = cv2.morphologyEx(binary, cv2.MORPH_OPEN, vertical_kernel)
    lines_mask = cv2.add(horizontal_lines, vertical_lines)
    return cv2.subtract(binary, lines_mask)
```

リスク: 漢字「一」「工」「口」など横・縦線の多い文字も除去されうる。カーネル長の閾値調整が必要。

現方針: 候補Aで検証し、枠線干渉が問題になった時点で候補Bを試す。

### Step 3-6. 輪郭検出（findContours）

`cv2.findContours` が二値画像の白い塊の境界を追跡し、各塊のバウンディングボックス・面積・頂点を返す。形状に関係なく閉じた輪郭を追跡するため、L字・多角形・穴あり形状にも対応できる。

---

### Step 3-7. 制約付き形状認識

ユーザーが形状のヒントを与えた場合、対応するフィット関数を直接呼ぶことで精度を安定させる。ヒントなしでは「これは楕円か四角か」をアルゴリズムが確定できない。

| ヒント | アルゴリズム | 概要 |
|---|---|---|
| 楕円 | `cv2.fitEllipse()` | 輪郭点群に最小二乗フィットした楕円を返す |
| 円 | `cv2.minEnclosingCircle()` | 全輪郭点を含む最小の外接円を返す |
| 長方形 | `cv2.minAreaRect()` | 面積最小の外接矩形（回転あり）を返す |
| 多角形（n角形） | `cv2.approxPolyDP()` | Douglas-Peuckerアルゴリズムで頂点を間引いてn角形に近似 |
| 自由曲線 | Bézierフィット | 輪郭点群から制御点を推定してBézier曲線で近似 |

---

### Step 3-8. 非矩形・非凸形状の認識

| 形状 | 対応難度 | 使うアルゴリズム |
|---|---|---|
| 長方形 | ★☆☆ | `minAreaRect()` |
| L字・凸でない多角形 | ★★☆ | `approxPolyDP()` |
| H形・コの字（単純連結） | ★★★ | `findContours` + 凹み検出 |
| 中に柱がある部屋（穴あり） | ★★★★ | `findContours` の階層構造 |

`findContours` の階層構造（`RETR_CCOMP` / `RETR_TREE`）を使うと、輪郭の親子関係から「外壁の中に穴がある」構造を取得できる。

### Step 4-1. テキスト→図形の距離計算（実装時に追記）

点と直線の距離: `|ax+by+c| / sqrt(a²+b²)`

---

### Step 4-2. 包含判定（Ray Casting）（実装時に追記）

多角形の内外判定。テキストが図形の内側に含まれるかを判定するために使う。

---

### Step 4-3. 近傍閾値の設計

「近傍」の閾値は実サンプルで実験的に決定する必要がある。手書き線が完全に閉じていない場合、包含判定が破綻するリスクもある。

## Step 3': 数式・座標解析（数式系・将来）

### 処理フロー

数式系では Step 3（図形抽出）・Step 4（空間関係推定）は使わない。OCR で読んだ文字を起点に意味を解析し、そこから図形を生成する。

**図形系との本質的な違い**

| | 図形系 | 数式系 |
|---|---|---|
| 出発点 | 画像上の図形 | OCR で読んだ文字（数式） |
| テキストの役割 | 図形に紐付くラベル | 意味そのもの |
| 図形の扱い | 画像から抽出する | 意味解析の結果として生成する |

手書きの直線だけ見ても `y=2x` か `y=1.8x` か判断できない。しかし文字として `y=2x` が読めれば、傾き・切片・グラフ形状まで確定する。数式系は「文字 → 意味解析 → 図形生成」の順で処理する。

```
Step 1 の text_blocks（OCR結果）
↓ 3'-1. 数式・座標の識別
↓ 3'-2. 数式パース
↓ 3'-3. 構造データへの変換
→ 出力層へ
```

---

### Step 3'-1. 数式・座標の識別（実装時に追記）

OCR結果の各テキストブロックが「数式・座標」か「通常テキスト」かを判定する。

---

### Step 3'-2. 数式パース（実装時に追記）

数式の種類ごとに解析方法が異なる。

| 入力例 | 種別 | 解析結果 |
|---|---|---|
| `y = 2x` | 線形関数 | 傾き=2, 切片=0 |
| `y = x²` | 二次関数 | 係数・次数 |
| `y = sin(x)` | 三角関数 | 関数種別・係数 |
| `A=(1,2), B=(3,4)` | 座標データ | 点ラベルと座標値の対 |
| `x²+4x+4=0` | 方程式 | 解の情報（x=-2） |

---

### Step 3'-3. 構造データへの変換（実装時に追記）

```json
// y = 2x の場合
{ "type": "linear_function", "slope": 2, "intercept": 0 }

// A=(1,2) の場合
{ "type": "point", "label": "A", "x": 1, "y": 2 }

// x²+4x+4=0 の場合
{ "type": "equation", "expression": "x**2+4*x+4", "solutions": [-2] }
```
