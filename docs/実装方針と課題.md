# GraphicCopy 実装方針・課題

最終更新: 2026-06-03

---

## 設計原則

- **出力は「構造データ」として設計する。SVGはその1出力に過ぎない**
- `shapes.py` / `relations.py` の出力フォーマットはSVGを知らない出力先非依存の形式にする（SVG特有のフィールドを持たせない）
- AIはOCR（認識）のみ担当・構造変換は決定論的アルゴリズムで処理
- GPU不要・CPUのみ・軽量・オフライン動作・スマホ対応
- まずPythonプロトタイプで核心アルゴリズムを検証し、その後Web/モバイルへ展開
- OCR用前処理と図形抽出用前処理は分離する（2026-05-18確定）
- 完全自動ではなく「ユーザーが形状ヒントを与えて精度を上げる」設計

---

## 確定した設計判断

### 対象外の決定

キャラクター・自由スケッチは対象外。Clip Studio Paint・Procreate等の既存ツールで解決済みの領域のため、GraphicCopyが勝てる領域ではない。

### OCRスタック

| 環境 | 使用モデル |
|---|---|
| スマホ / 軽量環境 | Tesseract OCR |
| 日本語手書き（必要な場合） | EasyOCR または専用CRNNモデル |
| PC 高精度オプション | TrOCR（標準構成には含めない） |

EasyOCRは「軽量」ではない（ResNet+CRNN構成）。TrOCRはスマホ500MB制限と矛盾するため標準構成から除外。

### 色除去を行わない理由

テキストが黒とは限らず、色でテキストと図形線を原理的に区別できない。有彩色テキスト（赤い注釈・青いラベルなど）がある場合に使えなくなるため。

### ヒントの渡し方（未決定）

- A: 処理前にパラメータで指定
- B: 手書き図の中にラベルを書く（OCRが読む）
- C: 処理後のUIで修正指定

どの方式が使いやすいかはプロトタイプ後に判断する。

### 間取り図の入力品質（未決定）

- A: ラフなナプキンスケッチ（壁が閉じていない、縮尺バラバラ）
- B: 丁寧な手書き図面（定規なし、プロポーション意識あり）
- C: 方眼紙・ノートの図（マス目活用、比較的正確）
- D: 既存の印刷済み間取り図の写真

**この決定が設計全体（ノイズ耐性・精度要件）に影響するため、プロトタイプ前に決める必要がある。**

## 既知の課題

| # | 課題 | 詳細 |
|---|---|---|
| 1 | **マスク生成の検証**（最優先） | 「OCRで読めた」と「文字を消せた」は別問題。未検証 |
| 2 | **「タ」「ー」未検出** | MSERで検出されていないかOCRの問題かが不明 |
| 3 | **矢印の誤検出** | 縦線が `\|` として検出される（フローチャート・回路図・間取り図全般で発生） |
| 4 | **空間関係推定の閾値** | 「近傍」の閾値は実サンプルで実験的に決定が必要。手書き線が閉じていない場合は包含判定が破綻するリスクもある |
| 5 | **透視補正の前提条件** | `warpPerspective`に紙の四隅座標が必要。自動検出は背景が複雑な場合に失敗しやすい |
| 6 | **日本語手書き対応の決断** | 英語のみ→Tesseractで開始可能。日本語手書きあり→最初からEasyOCRまたは専用モデルが必要 |
| 7 | **ユーザー修正フロー** | 誤認識テキストの編集・図形の差し替え手段がない。プロトタイプ段階では省略可だが、製品化を見据えると早期に設計が必要 |
| 8 | **矢印・有向エッジの扱い** | フローチャート・回路図での矢印を有向エッジとして認識するロジックが未定義 |
| 9 | **処理時間（9〜20秒）** | 精度優先で後回し |

## 将来の拡張方針

### Blender連携（保留）

ユーザーがBlender初心者のためモデリングワークフローの痛点が不明確。具体的に「ここが面倒」という場面が見えてから再検討する。

SVGの意味的構造（「ここが壁・空間・ドア」）が完成した時点でブーリアン・オフセットによる3D化が自然にできる設計になる。Blender連携は「後で追加する別機能」ではなく、SVGの意味的構造が完成した時点で自然に生えてくる拡張。**だからこそ今は2Dの清書に徹するのが正解。**

### ロフティングによる3D生成（記録のみ）

複数の2D断面図を積み重ねて3Dを生成するアプローチ。難所は「対応点の決定」（異なる形状の端同士をどう繋ぐか）。シンプルな形状同士（円・四角）なら解けるが、有機的な形状は困難。
