# GraphicCopy 実装方針・課題

最終更新: 2026-07-12

---

## 設計原則

- **出力は「構造データ」として設計する。SVGはその1出力に過ぎない**
- `shapes.py` / `relations.py` の出力フォーマットはSVGを知らない出力先非依存の形式にする（SVG特有のフィールドを持たせない）
- AIはOCR（認識）のみ担当・構造変換は決定論的アルゴリズムで処理
- GPU不要・CPUのみ・軽量・オフライン動作・スマホ対応
- まずPythonプロトタイプで核心アルゴリズムを検証し、その後Web/モバイルへ展開
- OCR用前処理と図形抽出用前処理は分離する（2026-05-18確定）
- 完全自動ではなく「ユーザーが形状ヒントを与えて精度を上げる」設計

## 確定した設計判断

### OCR検出パイプラインのCC刷新（2026-06-12確定）

Step 1 の検出系を MSERカスケードから**連結成分（CC）ベース**へ刷新する。確定設計・根因分析・移行計画・外部レビュー2巡の採否判断は `docs/fable 5提案.md`【確定版】を正とする。要点:

- 検出は「二値化 + CC」（二値化成功を前提に取りこぼしを構造的に防ぐ。二値化はインク率検査でfallback発動）
- CC分類は2パス4値（ノイズ/図形/文字候補/あいまい）。あいまいは行文脈で確定
- 行グルーピングは中心バンド方式（height_ratio条件は「ー」を単独クロップに追い込むため廃止）
- 切り抜きはCCピクセル描画方式（枠線リーク問題=候補A/B論争の解消）
- 信頼度ゲートは3帯域（採用≥70/保留50–70は複数CC行のみ/棄却<50）
- マスクは「採用」のみのピクセル精度（保留は含めない — 図形欠損は回復不能のため）
- **切替は評価ハーネスの数値で判断**（全サンプルで現行以上のときのみMSER削除）

### 評価ハーネスによる数値判断（2026-07-12確定）

目視のdebug画像を主たる判断材料にするのをやめ、`tests/eval_ocr.py` の数値で判断する。現行MSERパイプラインのベースラインは `tests/baseline.json` に固定済み（フロー.png: 検出再現率100% / 認識一致率66.7% / 誤検出1 / 処理時間6.0秒）。以降の変更はすべてこの数値との比較で採否を決める。

### 「タ」「ー」消失の原因確定（2026-07-12）

段階1の計装で決着した。**9文字すべてが検出フィルタを生き延びており、消失は認識系で起きていた**（詳細は `docs/開発ログ.md` 2026-07-12 の項）。

- 「ー」= **H4（OCR空振り脱落）**: 行マージ失敗 → 単独クロップ → 文脈なしOCRが空文字を返す
- 「タ」= **H5（新機序）**: `_deduplicate` が、Tesseractの返した幅広bbox（隣接文字を含む）を根拠に、隣の別文字を「重複」として除去

いずれも根本原因は**OCRの前に行を確定していないこと**であり、CC設計（行グルーピング → 行クロップ → PSM 7、NMS・dedup なし）で構造的に解消される。CC刷新の方針は変更なし。

### 対象外の決定

キャラクター・自由スケッチは対象外。Clip Studio Paint・Procreate等の既存ツールで解決済みの領域のため、GraphicCopyが勝てる領域ではない。

### OCRスタック

| 環境 | 使用モデル |
|---|---|
| スマホ / 軽量環境 | Tesseract OCR |
| 日本語手書き（必要な場合） | EasyOCR または専用CRNNモデル |
| PC 高精度オプション | TrOCR（標準構成には含めない） |

EasyOCRは「軽量」ではない（ResNet+CRNN構成）。TrOCRはスマホ500MB制限と矛盾するため標準構成から除外。

### 色除去を行わない理由

テキストが黒とは限らず、色でテキストと図形線を原理的に区別できない。有彩色テキスト（赤い注釈・青いラベルなど）がある場合に使えなくなるため。

### ヒントの渡し方（未決定）

- A: 処理前にパラメータで指定
- B: 手書き図の中にラベルを書く（OCRが読む）
- C: 処理後のUIで修正指定

どの方式が使いやすいかはプロトタイプ後に判断する。

### 間取り図の入力品質（未決定）

- A: ラフなナプキンスケッチ（壁が閉じていない、縮尺バラバラ）
- B: 丁寧な手書き図面（定規なし、プロポーション意識あり）
- C: 方眼紙・ノートの図（マス目活用、比較的正確）
- D: 既存の印刷済み間取り図の写真

**この決定が設計全体（ノイズ耐性・精度要件）に影響するため、プロトタイプ前に決める必要がある。**

## 既知の課題

| # | 課題 | 詳細 |
|---|---|---|
| 1 | マスク生成の検証 | **解決済み（2026-06-10）**。検出できたbboxは正しく白塗りされる |
| 2 | 「タ」「ー」未検出 | **原因確定（2026-07-12）**。「ー」=H4（単独クロップでOCR空振り）・「タ」=H5（`_deduplicate` が隣接文字を重複と誤認して除去）。CC設計で構造的に解消される見込み |
| 3 | **矢印の誤検出** | フロー.pngでは矢印が「し」(62%) として検出される。`CONFIDENCE_THRESHOLD = 0` のため素通り。3帯域ゲート（採用≥70）で棄却される見込み |
| 3b | **CC融合（枠＋矢印）** | 矢印は独立CCではなくノード枠と同一CCになっていることを実測で確認（2026-07-12）。CC設計の `empty_frame` 警告・再分割パスは現実に必要 |
| 4 | **空間関係推定の閾値** | 「近傍」の閾値は実サンプルで実験的に決定が必要。手書き線が閉じていない場合は包含判定が破綻するリスクもある |
| 5 | **透視補正の前提条件** | `warpPerspective`に紙の四隅座標が必要。自動検出は背景が複雑な場合に失敗しやすい |
| 6 | **日本語手書き対応の決断** | 英語のみ→Tesseractで開始可能。日本語手書きあり→最初からEasyOCRまたは専用モデルが必要 |
| 7 | **ユーザー修正フロー** | 誤認識テキストの編集・図形の差し替え手段がない。プロトタイプ段階では省略可だが、製品化を見据えると早期に設計が必要 |
| 8 | **矢印・有向エッジの扱い** | フローチャート・回路図での矢印を有向エッジとして認識するロジックが未定義 |
| 9 | **処理時間（9〜20秒）** | 精度優先で後回し |

## 将来の拡張方針

### Blender連携（保留）

ユーザーがBlender初心者のためモデリングワークフローの痛点が不明確。具体的に「ここが面倒」という場面が見えてから再検討する。

SVGの意味的構造（「ここが壁・空間・ドア」）が完成した時点でブーリアン・オフセットによる3D化が自然にできる設計になる。Blender連携は「後で追加する別機能」ではなく、SVGの意味的構造が完成した時点で自然に生えてくる拡張。**だからこそ今は2Dの清書に徹するのが正解。**

### ロフティングによる3D生成（記録のみ）

複数の2D断面図を積み重ねて3Dを生成するアプローチ。難所は「対応点の決定」（異なる形状の端同士をどう繋ぐか）。シンプルな形状同士（円・四角）なら解けるが、有機的な形状は困難。
