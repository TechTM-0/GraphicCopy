## 2026-05-10 頃 — プロジェクト開始・基本方針確定

**変更概要**: GraphicCopy プロジェクト発足。基本設計方針を確定。

**確定内容**:
- 出力は「構造データ」として設計する。SVGはその1出力に過ぎない
- AIは認識（OCR）のみ担当、構造変換は決定論的アルゴリズムで処理
- GPU不要・CPUのみ・軽量・オフライン動作・スマホ対応
- まず Python プロトタイプで核心アルゴリズムを検証し、その後 Web/モバイルへ展開

## 2026-05-18 — OCR前処理と図形抽出前処理の分離確定

**変更概要**: Step 1（OCR）用の前処理と Step 3（図形抽出）用の前処理を別モジュールに分離する方針を確定。

**根拠**: OCR は文字の読みやすさ（コントラスト・拡大）を優先するが、図形抽出はエッジの保存を優先する。同一前処理では両方の精度が中途半端になる。

**変更概要**: `shapes.py` / `relations.py` の出力フォーマットを SVG 非依存の構造データとして定義する方針を確定。

## 2026-05-10〜06-09 — OCR 手法の試行錯誤（フロー.png を対象に検証）

**背景**: Tesseract に全体画像を渡しても小さいテキスト（10〜14px）を見つけられないことが判明。問題は OCR の読解能力ではなく、「どこを切り抜くか」を見つけること。

### 試行1: 全体画像 OCR（PSM 11）

| 項目 | 内容 |
|---|---|
| 設定 | グレースケール変換のみ、Tesseract PSM 11（スパーステキスト） |
| 結果 | 「スタート」○、「初期化」「検証」× |
| 処理時間 | 約2秒 |
| 失敗原因 | 小領域（10〜14px）を全体画像から見つけられない |

### 試行2: PSM 変更（PSM 3・PSM 6）

| 項目 | 内容 |
|---|---|
| 設定 | PSM 3（自動セグメンテーション）・PSM 6（均一ブロック想定）に変更 |
| 結果 | PSM 11 より悪化 |
| 失敗原因 | PSM 変更は「領域を見つけられない」根本問題を解決しない |

### 試行3: タイル分割（500px・50% 重複）

| 項目 | 内容 |
|---|---|
| 設定 | 画像を 500px タイルに分割し各タイルを個別 OCR |
| 結果 | 「スタート」×、「初期化」「検証」× |
| 処理時間 | 15秒 |
| 失敗原因 | 枠線のみのタイルで Tesseract が線を文字として誤読。ノイズだらけになる |

### 試行4: MSER（gap=10・scale=2倍）

| 項目 | 内容 |
|---|---|
| 設定 | `cv2.MSER` で文字候補領域を検出。gap=10、全体画像2倍拡大 |
| 結果 | 「スタート」△、「初期化」は「初」「化」のみ（「期」脱落） |
| 処理時間 | 1.8秒 |
| 失敗原因 | gap が小さすぎて「期」のように内部空間のある文字が安定領域として検出されない |

### 試行5: MSER（gap=30・scale=4倍）

| 項目 | 内容 |
|---|---|
| 設定 | gap=30 に拡大、全体画像4倍拡大 |
| 結果 | 「スタート」×、「初期化」「検証」× |
| 処理時間 | 0.1秒 |
| 失敗原因 | gap が大きすぎて全領域が連鎖し1つの巨大 bbox に合体してしまう |

### 試行6: MSER + NMS（現行実装として採用）

| 項目 | 内容 |
|---|---|
| 設定 | `_min_area=150, _max_area=14400`、全体画像4倍（INTER_CUBIC）拡大後 MSER、IoU > 0.3 で NMS（面積優先）、height_ratio ≥ 0.8・y_overlap ≥ 0.7・gap ≤ max_h の3条件で横方向マージ |
| 検出件数推移 | MSER 472件 → NMS後 18件 → マージ後 9件 → OCR後 10件 |
| 結果 | 「ス」「ト」検出、「タ」「ー」未検出。「初期化」「検証」全文字 95〜96% で検出 |
| 処理時間 | 9.3秒（最大 20秒） |
| 観察された問題 | ① 「ス＋タ」をまたぐキメラ bbox が NMS で残存（debug_bbox.png で確認）② 「タ」「ー」の bbox が最終出力に存在しない ③ 矢印（↓）が 62% 信頼度で誤検出 ④ CONFIDENCE_THRESHOLD = 0 のため全信頼度が素通り |

**確認できた事実（試行6 終了時点）**:
- 「スタート」: グレースケール + Otsu + 2倍拡大で文字ごとに検出できる（各 96%）
- 「初期化」: 該当領域を切り抜いて OCR すると検出できる（各 95%以上）
- 問題は OCR の読解能力ではなく、全体画像から小さいテキスト領域を見つけること

## 2026-06-12 — CC パイプライン設計確定・docs 全面更新

**変更概要**: MSER カスケードの根因分析を実施し、連結成分（CC）ベースへの刷新設計を確定。外部レビュー2巡を経て `docs/fable 5提案.md` として確定版を作成。

### 根因分析の結論

| 問題 | 根因 |
|---|---|
| 「タ」「ー」消失 | 下記4仮説のいずれか（段階1の計装で確定） |
| H1: MSER 未検出 | 細ストロークが安定領域にならない（疎なカタカナに不向き） |
| H2: フィルタ足切り | サイズフィルタ（min_h=20・4倍スケール）またはアスペクト比フィルタ |
| H3: NMS キメラ吸収 | 面積優先 NMS が「ス＋タ」複合領域を残し個別 bbox を抑制 |
| H4: OCR 空振り脱落 | height_ratio 失敗→「ー」単独クロップ→文脈なし OCR が空文字を返す |
| 処理時間 9〜20秒 | 全体画像を4倍 INTER_CUBIC 拡大した後 MSER を全面実行 |
| 枠線リーク | PAD=10px の余白に枠線ピクセルが混入し Tesseract のセグメンテーションが乱れる |
| キメラ bbox | NMS の「面積最大優先」が複数文字をまたぐ複合領域を優先して残す |

### 確定した新パイプライン（A〜G）

| ステップ | 内容 | 旧実装からの変更 |
|---|---|---|
| A | min(B,G,R) → Otsu + インク率検査（fallback: Adaptive） | グレースケール変換を輝度式から min に変更。fallback を「存在しなかったカスケード」から「インク率検査」に修正 |
| B | CC 分析（1倍スケール） | 4倍全画像拡大 + MSER を廃止 |
| C | CC 2パス分類（ノイズ/図形/文字候補/あいまい） | MSER サイズフィルタ・アスペクト比フィルタを廃止。文字サイズ自動推定で固定閾値を排除 |
| D | 中心バンド行グルーピング + あいまい CC 確定 | height_ratio ≥ 0.8 を廃止。`｜cy(CC) − cy(行)｜ ≤ 0.4 × 行高さ` に変更 |
| E | CC ピクセル描画による行クロップ生成 | PAD 余白方式を廃止。枠線リーク問題を構造的に解消（候補 A/B 論争に終止符） |
| F | 行クロップのみ高さ正規化（≈40px） + Tesseract PSM 7 | 全体画像拡大を廃止。クロップのみ拡大 |
| G | 信頼度3帯域ゲート（採用≥70/保留50〜69は複数CC行のみ/棄却<50） | CONFIDENCE_THRESHOLD = 0（全通過）を廃止 |
| Mask | 採用 CC のピクセル + 2〜3px 膨張 | bbox 矩形マスクを廃止。CCピクセル精度に変更。保留はマスクに含めない |

## 2026-07-12 — 段階1: 評価ハーネス＋段階別計装＋ベースライン計測（[#1](https://github.com/TechTM-0/GraphicCopy/issues/1)）

**変更ファイル**: `src/graphiccopy/ocr.py`（計装追加）・`tests/eval_ocr.py`（新規）・`samples/flow/フロー.expected.json`（新規）・`tests/test_ocr.py`（glob再帰化）

**変更内容**:

- `run_ocr` / `run_ocr_with_preprocess` に省略可能な `debug: dict` を追加。`None` のときは従来と完全に同一動作。各段階のbbox・OCRに渡した切り抜き・**Tesseractの生出力（空文字を含む）**・OCR呼び出し回数を記録する
- `tests/eval_ocr.py`: 検出再現率 / 認識一致率 / 誤検出数 / 処理時間 / OCR呼び出し回数 / 空振り回数を計測。認識一致率は正規化編集距離（`1 - lev/len`）で部分点を付ける（完全一致判定だと「スート」と空文字が同点になり改善が見えないため）
- 文字別の段階追跡表を出力。判定は「GT文字の中心が段階のbboxに含まれるか」（IoU閾値では、MSERが1文字を分割した部分検出を未検出と誤判定するため）
- 正解データは Otsu二値化後の連結成分から実測。**「ー」の実体は 36×5px**（他のカタカナは 35×33〜39px）
- `samples/` をドメイン別サブフォルダ化（`samples/flow/`）

**計測結果（現行MSERパイプラインのベースライン・`tests/baseline.json` に固定）**:

| 指標 | フロー.png |
|---|---|
| 検出再現率 | 100.0%（3行すべてbbox取得） |
| 認識一致率 | 66.7% |
| 誤検出数 | 1（矢印を「し」62%として検出） |
| OCR呼び出し回数 | 9 |
| 空振り回数 | 1 |
| 処理時間 | 平均 6.0秒 |
| 段階別bbox件数 | MSER 472 → サイズ 419 → NMS 18 → マージ 9 → アスペクト 9 |
| 行別認識 | スタート→`スト` / 初期化→`初期化` / 検証→`検こ・証` |

**「タ」「ー」消失段階の確定（H1〜H4の決着）**:

**9文字すべてがアスペクトフィルタまで生存**していた。よって **H1（MSER未検出）・H2（フィルタ足切り）は棄却**。着手前の予測（「ー」は h=5px なのでサイズフィルタで落ちる）は**外れ**。犯人は2つとも検出系ではなく認識系にあった。

| 文字 | 消失段階 | 機序 |
|---|---|---|
| ー | **H4（OCR空振り脱落）** | マージ条件 `height_ratio ≥ 0.8` が「ー」(h=5) と「ス」(h=33) を結合できず（比0.15）、`crop_05` として**単独クロップ**でOCRに送られ、Tesseractが空文字を返した。fable 5提案 節1-3 の間接経路の記述どおり |
| タ | **H3の変種（新仮説H5）** | NMSキメラ（ス＋タ）クロップ `crop_01` に対し、Tesseractは **ス(95%)・タ(82%) の両方を返していた**。しかし「ス」のbboxが幅317px（クロップ幅343の92%）で「タ」を包含 → `_deduplicate` が IoU=0.45 > 0.3 で「タ」を**同一テキストの重複と誤認して除去** |

**観察・判断**:

- H5（`_deduplicate` による隣接文字の誤除去）は fable 5提案 の4仮説になかった**第5の機序**。`_deduplicate` は「同一テキストの重複除去」のつもりの関数だが、Tesseractが日本語で隣接文字を含む広いbboxを返すため、隣接する別文字を消す凶器になっていた
- 根本原因はいずれもキメラクロップと単独クロップ、すなわち**「OCRの前に行を確定していない」こと**。CC設計（行グルーピング → 行クロップ → PSM 7）は NMS も `_deduplicate` も持たないため、H4・H5の両方が構造的に消滅する
- **CC融合が実在**: 矢印（↓）は独立CCではなく**ノード枠と同一CC**（CC#6・#12）。fable 5提案 節2の `empty_frame` 警告と再分割パスの入口は現実に必要
- 誤検出「し」(62%)・ノイズ「こ」(56%)・「・」(31%) はすべて `CONFIDENCE_THRESHOLD = 0` を素通りしている。3帯域ゲート（採用≥70）で棄却される見込み
- 「初期化」は5CC・「検証」は断片含む5CCに分かれる。CC設計の文字サイズ自動推定と行グルーピングが前提どおり必要
- **次のアクション**: 段階2（`detect.py` 新規実装・A〜D）へ進む
