## 管理ルール（正本の置き場）

**1つの事実は1か所にしか置かない。** 置き場は「**不変か・可変か**」で決める。

| 性質 | 正本 | 例 |
|---|---|---|
| **不変**（一度確定したら書き換わらない） | **GitHub Issue のコメント** | 試行錯誤の記録・計測結果・観察・**なぜそう決めたかの根拠**・レビューの採否・撤回の経緯 |
| **可変・コードと同期が必要** | **`docs/`** | 現在のアルゴリズム（`ロジック解説.md`） |
| **可変・コードと同期不要** | **GitHub Issue の本文**（上書きする） | 現在有効な設計判断・未決定事項・既知の課題（#9） |
| **タスクの一覧と現在地** | **GitHub Issue** ＋ `open-tasks.md`（リンクのみのダッシュボード） | 段階1〜5・残課題 |

**なぜこの基準か**: バージョン管理は「変わるもの」の変化を追う仕組みなので、**不変の情報をgit管理しても得るものがない**。逆にアルゴリズム解説はコードと一緒に変わるため、同じコミットで動かないと乖離する。

### Issue の使い分け（#9 が典型）

- **本文** = 現在有効な結論。変わったら**上書きする**
- **コメント** = そこに至った経緯・根拠（**追記のみ・書き換えない**）

### 相互リンクの義務

- 設計を変える判断は**Issueのコメントで決着させ**、結論をIssue本文に反映する
- コードを変えたら、Issueコメントに**コミットハッシュを貼る**
- docs側には判断の出典としてIssue番号を書く（例: `（2026-06-12確定 → #2）`）

### タスク完了時にやること

1. Issueに実装内容・計測結果・観察・判断をコメントする（`gh issue comment`）
2. アルゴリズムが変わったなら `docs/ロジック解説.md` を更新し、**同じコミットに含める**
3. 設計判断が変わったなら **#9 の本文を上書き**し、根拠をコメントに追記する
4. Issueをクローズし、`open-tasks.md` の現在地・次の一手・残タスクを更新する
